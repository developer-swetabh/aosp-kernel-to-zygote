⚙️ Linux Kernel

Decompresses into memory, initializes hardware, and mounts the initial ramdisk — the bridge from bootloader to userspace.

📋 Overview

  • Purpose: Initialize the hardware platform and provide the foundation for all Android userspace processes.
  • GKI (Generic Kernel Image): Since Android 12, a single Google-maintained kernel binary runs across all devices, with vendor modules loaded separately.
  • Output: Mounts initramfs as root and executes /init (PID 1) — the first userspace process.

⚙️ How It Works

Kernel Decompression

The compressed kernel image (LZ4/gzip) is decompressed by the bootloader or self-extracting stub. The uncompressed kernel is placed at a fixed physical address determined by the SoC memory map.

Architecture-Specific Initialization

CPU mode setup (EL1 for ARM64), MMU page tables initialized, interrupt controller configured. Device-tree blob (DTB) is parsed to discover hardware topology.

Core Subsystem Init

Scheduler, memory management (SLUB), VFS, block layer, and network stack are initialized. Kernel command-line arguments (androidboot.*) are parsed and stored.

Driver Initialization

Built-in drivers probed in dependency order. GKI modules (*.ko) are loaded from the vendor ramdisk. Key early drivers: eMMC/UFS storage, PMIC, clock controllers.

Mount initramfs & exec /init

The initial ramdisk (embedded in boot.img or init_boot.img) is unpacked as a tmpfs root filesystem. Kernel calls kernel_init() → run_init_process("/init"), creating PID 1.

⚠️ Common Checks & Failure Modes

🚫 Kernel Panic on Boot

Missing critical driver or wrong DTB → kernel panics during init. Check /proc/last_kmsg or serial console output.

🚫 initramfs Not Found

If the ramdisk address or size is wrong, kernel fails to mount root. Error: "VFS: Cannot open root device".

⚡ Module Load Failure

GKI module version mismatch → module refuses to load. Check vermagic string matches running kernel.

⚡ DTB Mismatch

Wrong device tree for the hardware → drivers probe incorrectly, system unstable. Board ID in DTB must match SoC.

🏭 Vendor-Specific Details

🔧 GKI vs Vendor Kernel

GKI (Android 12+): Google ships a single kernel binary per architecture. Vendor-specific code is in loadable kernel modules (*.ko).

Legacy: Before GKI, each vendor shipped a fully custom kernel fork, making updates difficult.

GKI enforces a stable Kernel Module Interface (KMI) — vendor modules must be ABI-compatible.

🔧 Qualcomm kernel specifics

Qualcomm BSP adds a vendor ramdisk containing QTI-specific modules: qcom-cpufreq-hw.ko, phy-qcom-ufs.ko, etc.

DTB overlays in dtbo.img customize the base DTB for specific Qualcomm SKUs.
