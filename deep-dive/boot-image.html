📦 Boot Image Format Evolution

From one monolithic image to a clean split — how Android 13 reorganized the boot partition for modularity, faster updates, and a better engineering experience.

📋 Overview

  • What changed: Android 13 (Tiramisu) split the classic boot.img into two images — boot.img (kernel only) and init_boot.img (generic ramdisk) — completing a modularization journey that began with GKI in Android 11.
  • Why it matters: This separation allows Google to update the kernel independently from the ramdisk, and vice versa — enabling smaller OTAs, faster security patches, and cleaner architecture.
  • Key insight: The boot image is the "delivery envelope" for the kernel and ramdisk. Understanding its format helps you reason about OTA updates, verified boot, and partition layout.

🎯 Think of It This Way

Imagine you're shipping two items — a laptop (kernel) and a power adapter (ramdisk). In the old days, Android packed both into one sealed box (boot.img). If you needed to replace just the adapter, you had to repackage and reship the entire box — laptop included.

Android 13 changed this: now the laptop ships in its own box (boot.img) and the adapter ships in its own box (init_boot.img). Need to update the adapter? Ship just that small box. The laptop box stays untouched. Simple, modular, efficient.

💾 The Generic Boot Partition

The boot partition is one of the most critical partitions on an Android device. It's what the bootloader (ABL) reads first after selecting the active A/B slot. Its contents determine what kernel runs and what initial userspace environment is available before the real filesystem is mounted.

As a junior engineer, here's the key mental model: the boot partition is the launchpad. Without it, the device has no kernel to run and no ramdisk to bootstrap Android. Everything else — mounting /system, starting services, launching apps — depends on what this partition delivers.

The boot partition always contains a header that tells the bootloader where each component starts and how large it is. The header version has evolved over time as Android's architecture matured — let's trace that evolution.

📅 Boot Image Header Versions

Each header version reflects an architectural shift. Understanding these helps you read boot.img headers and diagnose partition issues.

Header v0 — Android 9 and earlier

Original format. Single image containing: kernel + generic ramdisk + second-stage bootloader (optional). Everything in one package — simple but inflexible.

Header v1 — Android 10 (Q)

Added recovery DTBO section for devices using recovery-as-boot. Maintained backward compatibility with v0 layout for kernel and ramdisk.

Header v2 — Android 10 (Q)

Added DTB (Device Tree Blob) section to boot.img. Now boot.img could carry hardware descriptions. Still a monolithic image.

Header v3 — Android 11 (R) 🔑 Key change

The GKI revolution begins. Simplified header. Removed second-stage bootloader, recovery DTBO, and DTB from boot.img — those moved to vendor_boot.img (new partition). Page size fixed to 4096. Boot.img now carries only: kernel + generic ramdisk.

Header v4 — Android 12 (S) → Android 13+ (T)

Same as v3 for boot.img, but vendor_boot.img gained a vendor ramdisk fragment table for multi-vendor ramdisk support. Then in Android 13, the generic ramdisk was moved to a new init_boot.img partition — boot.img now contains only the kernel.

Notice the trend: each version removes something from boot.img and gives it its own home. By Android 13, boot.img has been slimmed down to just the kernel — everything else lives in dedicated partitions. This is modularization in action!

🤔 Why the Change? Architectural Motivations

GKI: Google's Kernel Independence

Starting with Android 11, Google introduced the Generic Kernel Image (GKI) — a single kernel binary maintained by Google, replacing hundreds of vendor-forked kernels. But the GKI kernel was still packaged alongside the generic ramdisk in boot.img, limiting how independently each could be updated.

The Coupling Problem

Imagine Google ships a kernel security patch. With the old layout, OEMs had to rebuild boot.img including the ramdisk (which they may have customized). Two independent components forced through the same pipeline — slower, more error-prone, and larger OTA downloads.

vendor_boot.img Paved the Way

Android 11's vendor_boot.img already demonstrated the split principle by moving vendor-specific ramdisk content out of boot.img. It proved the concept worked. The logical next step was to also separate the generic ramdisk.

Treble's Full Vision

Project Treble (Android 8+) aimed to separate the Android framework from vendor code. The boot image split completes this vision at the boot level: Google controls boot.img (kernel) and init_boot.img (generic ramdisk), while OEMs control vendor_boot.img (vendor ramdisk).

The Result: init_boot.img (Android 13)

Android 13 introduced the init_boot partition. The generic ramdisk — containing /init (first-stage init), basic init.rc fragments, and recovery resources — now lives in its own image. Clean boundaries, independent lifecycles, modular updates.

🧩 Image Composition — What's Inside Each Image?

📦 boot.img (Android 13+)
  • Boot header v4 — magic number, header size, kernel size/offset, OS version
  • GKI kernel — compressed kernel binary (LZ4 or gzip), ~30–40 MB
  • Kernel command line — androidboot.* parameters (hardware, slot suffix, etc.)
  • No ramdisk — generic ramdisk has moved to init_boot.img
✨ init_boot.img (Android 13+)
  • Boot header v4 — same header format as boot.img
  • Generic ramdisk containing:
    • /init — first-stage init binary (PID 1)
    • init.rc fragments — early service definitions
    • Recovery resources — GSI (Generic System Image) uses these for recovery mode
    • /system/etc/ramdisk/ — additional first-stage config files
  • Typically ~5–15 MB
🏭 vendor_boot.img (since Android 11)
  • Vendor boot header — vendor ramdisk size, DTB size, vendor cmdline
  • Vendor ramdisk — loadable kernel modules (*.ko), vendor-specific init.*.rc scripts, fstab.<hw>
  • DTB (Device Tree Blob) — hardware description for the kernel
  • Vendor ramdisk fragment table (v4) — allows multiple vendor ramdisk fragments for different use cases
  • Entirely OEM-controlled — Google does not modify this image
Here's how to remember the ownership: Google controls boot.img (kernel) and init_boot.img (generic ramdisk). OEMs control vendor_boot.img (vendor ramdisk + hardware descriptions). This clean ownership boundary is what makes independent updates possible.

🔨 AOSP Build System Decisions

How does the AOSP build system decide whether to produce init_boot.img? It all comes down to a few BoardConfig.mk variables:

VariableValueEffect
BOARD_BOOT_HEADER_VERSION 4 required Sets boot image header to v4. Required for Android 13+ devices.
BOARD_INIT_BOOT_HEADER_VERSION 4 required When set, the build system produces init_boot.img containing the generic ramdisk. The generic ramdisk is excluded from boot.img.
BOARD_RAMDISK_USE_LZ4 true optional Compress the ramdisk with LZ4 (faster decompression) instead of gzip.
BOARD_USES_GENERIC_KERNEL_IMAGE true required Signals that the device uses GKI. Required for init_boot.img generation.
BOARD_VENDOR_BOOT_HEADER_VERSION 4 required Sets the vendor boot image header version. v4 adds vendor ramdisk fragment table.
makefile 
# Example BoardConfig.mk for an Android 13+ device
# This configuration produces: boot.img + init_boot.img + vendor_boot.img

BOARD_BOOT_HEADER_VERSION := 4
BOARD_INIT_BOOT_HEADER_VERSION := 4
BOARD_VENDOR_BOOT_HEADER_VERSION := 4
BOARD_USES_GENERIC_KERNEL_IMAGE := true
BOARD_RAMDISK_USE_LZ4 := true

# Partition sizes
BOARD_BOOTIMAGE_PARTITION_SIZE := 67108864          # 64 MB
BOARD_INIT_BOOT_IMAGE_PARTITION_SIZE := 8388608     # 8 MB
BOARD_VENDOR_BOOTIMAGE_PARTITION_SIZE := 67108864   # 64 MB
The key trigger is BOARD_INIT_BOOT_HEADER_VERSION. When this variable is defined, the AOSP build system (build/make/core/Makefile) routes the generic ramdisk into init_boot.img instead of packing it into boot.img. If it's not set, you get the legacy behavior — everything in boot.img.
📂 Where does the build logic live?

The key build rules are in build/make/core/Makefile. Search for INSTALLED_INIT_BOOT_IMAGE_TARGET to find where init_boot.img is assembled. The mkbootimg tool (in system/tools/mkbootimg/) is responsible for constructing the actual binary image from the kernel, ramdisk, and header components.

You can also look at build/make/core/board_config.mk for the validation logic that checks variable consistency (e.g., BOARD_INIT_BOOT_HEADER_VERSION requires BOARD_USES_GENERIC_KERNEL_IMAGE to be true).

🔄 Modern Boot Chain Flow

Now let's see how the bootloader actually uses these three images at boot time. This is the step-by-step flow on an Android 13+ device:

ABL selects active slot (A or B)

The bootloader reads slot metadata from the misc partition to determine which slot is healthy. Suppose it selects slot A. All partition names are now suffixed with _a.

ABL reads boot_a partition → loads boot.img

Parses the v4 header. Extracts the compressed kernel into RAM. Reads the kernel command line. Notes that the ramdisk size in the header is zero — the bootloader knows it needs to look elsewhere for the ramdisk.

ABL reads init_boot_a partition → loads init_boot.img

Parses the header. Extracts the generic ramdisk (containing /init and core init.rc fragments) into a RAM buffer.

ABL reads vendor_boot_a partition → loads vendor_boot.img

Extracts the vendor ramdisk (kernel modules, vendor init scripts, fstab). Also reads the DTB (device tree blob) and vendor kernel command-line additions.

ABL concatenates ramdisks

The bootloader overlays the vendor ramdisk on top of the generic ramdisk in memory, creating a single combined ramdisk. Vendor files override generic files where paths overlap. The combined command line is also assembled.

ABL decompresses kernel and jumps to entry point

The kernel begins execution. It finds the combined ramdisk in memory (address passed via boot params), mounts it as initramfs, and executes /init — the first-stage init that came from init_boot.img. The boot chain continues as usual from here.

A handy analogy for the ramdisk merge: think of the generic ramdisk as a base layer (like a Docker base image) and the vendor ramdisk as an overlay layer. The bootloader stacks them in memory: generic first, vendor on top. Vendor files win when there's a path conflict. The kernel sees only the final merged result.

🎁 Practical Benefits

🏭 For OEMs

Update the vendor ramdisk (drivers, vendor init scripts) without touching the Google-maintained kernel or generic ramdisk. Reduces integration effort during quarterly platform updates.

⚙️ For Google

Ship GKI kernel updates directly to devices without coordinating ramdisk changes with OEMs. Enables monthly kernel security patches independent of the platform release cycle.

👨‍💻 For Developers

Smaller OTA deltas — changing only the kernel means downloading ~30 MB instead of ~45 MB. Faster local build-flash-test cycles when iterating on ramdisk changes.

👤 For Users

More frequent security updates (kernel patches arrive independently). Smaller update downloads. Less downtime during OTA installs.

⚠️ Common Checks & Failure Modes

🚫 init_boot partition missing

If the device's partition table doesn't have init_boot_a/b but the build system produced init_boot.img, flashing will fail. Ensure the partition table (GPT) is updated before flashing a 13+ build on older hardware.

🚫 Header version mismatch

If BOARD_BOOT_HEADER_VERSION doesn't match what the bootloader expects, ABL may fail to parse the image. Symptom: device stuck at bootloader or bootloop.

⚡ Ramdisk in wrong image

If BOARD_INIT_BOOT_HEADER_VERSION is set but you manually pack a ramdisk into boot.img, the bootloader may use the wrong ramdisk (or skip init_boot entirely). Always let the build system handle image composition.

⚡ AVB hash descriptor mismatch

After manually modifying init_boot.img, the AVB hash descriptor in vbmeta won't match. On locked devices this causes boot failure. Re-sign with avbtool or use --disable-verity during development.

🏭 Vendor-Specific Details

🔧 Qualcomm devices

Qualcomm's ABL (Android Bootloader running as a UEFI application inside XBL) has supported the init_boot partition since its Android 13 BSP releases. The ABL reads three partitions sequentially: boot_a/b, init_boot_a/b, vendor_boot_a/b.

Qualcomm's BSP typically sets BOARD_INIT_BOOT_HEADER_VERSION := 4 in device/qcom/<target>/BoardConfig.mk. The fstab.qcom lives in the vendor ramdisk (vendor_boot.img).

🔧 Google Pixel (Tensor)

Pixel 7 and later (Tensor G2+) were among the first devices to ship with init_boot.img. Google uses these devices as the reference implementation. Pixel factory images clearly show the three-image split in their flash packages.

🔧 Devices upgrading to Android 13

Devices that launched with Android 12 or earlier and are upgrading to Android 13 are not required to adopt init_boot.img. The split is only mandatory for devices launching with Android 13+. Upgrading devices can continue using the legacy boot.img layout with the generic ramdisk included.

This is an important distinction: the requirement applies to launch API level, not the current running API level.

❓ What about recovery?

With the split, recovery resources (used by the Generic System Image for A/B recovery) are included in init_boot.img's ramdisk. Devices using a dedicated recovery partition may handle this differently. The exact behavior depends on whether the device uses BOARD_USES_RECOVERY_AS_BOOT or a separate recovery partition.

Uncertainty note: The interaction between init_boot.img and dedicated recovery partitions can vary by vendor implementation. Consult your vendor's BSP documentation for specifics.

💻 Code Snippets

bash 
# Inspect a boot image header (works for boot.img and init_boot.img)
unpack_bootimg --boot_img boot.img --out out_boot/
cat out_boot/boot_img_hdr     # Shows header version, kernel size, ramdisk size

# Inspect init_boot.img
unpack_bootimg --boot_img init_boot.img --out out_init_boot/
ls -la out_init_boot/          # Should show ramdisk but no kernel

# Create a boot.img from components (Android 13+ style, no ramdisk)
mkbootimg \
  --header_version 4 \
  --kernel Image.lz4 \
  --cmdline "androidboot.hardware=qcom" \
  --output boot.img

# Create init_boot.img
mkbootimg \
  --header_version 4 \
  --ramdisk ramdisk.cpio.lz4 \
  --output init_boot.img
bash 
# Check which images exist on a running device
adb shell ls -la /dev/block/by-name/ | grep -E "boot|init_boot|vendor_boot"
# Expected output on Android 13+ device:
# boot_a -> /dev/block/sdXN
# boot_b -> /dev/block/sdXN
# init_boot_a -> /dev/block/sdXN     ← This confirms init_boot partition exists
# init_boot_b -> /dev/block/sdXN
# vendor_boot_a -> /dev/block/sdXN
# vendor_boot_b -> /dev/block/sdXN

# Flash individual images via fastboot
fastboot flash boot boot.img
fastboot flash init_boot init_boot.img
fastboot flash vendor_boot vendor_boot.img
bash 
# Extract and inspect the generic ramdisk contents from init_boot.img
unpack_bootimg --boot_img init_boot.img --out /tmp/init_boot_out
cd /tmp/init_boot_out
lz4 -d ramdisk /tmp/ramdisk.cpio      # Decompress if LZ4
mkdir /tmp/ramdisk_contents
cd /tmp/ramdisk_contents
cpio -idv < /tmp/ramdisk.cpio         # Extract cpio archive
ls -la                                  # Should contain: init, init.rc, system/, etc.
