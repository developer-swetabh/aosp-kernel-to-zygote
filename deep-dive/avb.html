🛡️ AVB — dm-linear & dm-verity Deep Dive

How Android carves logical partitions from the super block device with dm-linear, verifies every block with dm-verity, and ties it all together through Android Verified Boot (AVB) — the end-to-end flow from raw disk to trusted mount.

📋 Overview

  • dm-linear is a kernel device-mapper target that exposes a contiguous byte range of a larger block device as its own virtual block device. On Android, it carves logical partitions (system, vendor, odm…) out of the single physical super partition. (AOSP — Dynamic Partitions)
  • dm-verity is a device-mapper target that provides transparent, read-time integrity checking. It uses a Merkle hash tree to verify every 4 KiB block as it is read. The dm-verity target wraps an underlying block device — often a dm-linear device. (AOSP — dm-verity)
  • AVB (Android Verified Boot 2.0) is the orchestration layer. It reads signed vbmeta structures, verifies cryptographic signatures, checks rollback indices, and then instructs fs_mgr to create the dm-linear and dm-verity targets so partitions can be mounted as trusted. (AOSP — AVB 2.0)

🎭 Think of It This Way

Imagine a large warehouse (the super partition) full of sealed crates. dm-linear is the forklift that pulls out a specific crate by its shelf coordinates — it does not inspect the contents, it only delivers the right block range. dm-verity is the tamper-evident seal on that crate: every time someone opens a panel (reads a 4 KiB block), the seal verifies the panel has not been swapped or altered. AVB is the security office that first checks the warehouse manifest is authentically signed (vbmeta), then authorises the forklift (dm-linear) and attaches the seals (dm-verity) before the crate is delivered to the loading dock (mount point).

Without the forklift, you cannot reach the crate. Without the seal, you cannot trust the contents. Without the security office, anyone could forge the manifest.

⚙️ End-to-End Flow — Super to Verified Mount

① Super Partition Layout

Modern Android devices (Android 10+) use a single physical partition called super. Inside it, a metadata header (the LP metadata, typically at the first 4 KiB) describes the layout: which byte ranges belong to system, vendor, odm, product, etc. Each range is called an extent. A logical partition can span one or more extents.

Before dynamic partitions, each logical partition was a separate physical partition sized at build time. Resizing required repartitioning. Dynamic partitions let OEMs resize logical partitions in OTA updates because they are just metadata entries inside super.
📄 AOSP — Dynamic Partitions · liblp (LP metadata parser)

② Carving — Create dm-linear Target

During first-stage init, fs_mgr reads the LP metadata from super and, for each logical partition marked first_stage_mount in fstab, creates a dm-linear device-mapper target. The dm-linear mapping table looks like:

0 <size_in_sectors> linear /dev/block/by-name/super <start_sector>

This tells the kernel: “Create a virtual block device whose sector 0 maps to sector start_sector of the physical super device, and spans size_in_sectors sectors.” The result is a device node like /dev/block/dm-0.

dm-linear performs no verification and no transformation. It is a pure address translator. Think of it as a window cut into the larger block device. Every read at offset X on the dm-linear device becomes a read at offset (start + X) on super.
📄 Kernel docs — dm-linear · fs_mgr_dm_linear.cpp

③ AVB Verification — vbmeta Signature & Rollback

Before setting up dm-verity, fs_mgr (via libavb) reads the vbmeta partition (and any chained vbmeta like vbmeta_system, vbmeta_vendor). For each:

1. Verify the RSA signature against the embedded or OEM-provided public key.
2. Check the rollback index against the minimum stored in RPMB (Replay-Protected Memory Block) — reject if the image is older.
3. Extract descriptors: hash descriptors (for small partitions like boot), hashtree descriptors (for large partitions like system), and chain descriptors (pointers to child vbmeta).

This step establishes the cryptographic root of trust. The hashtree descriptor for system contains the Merkle tree root hash — the single value that anchors integrity for every block on the partition. If the vbmeta signature is invalid, the entire chain is untrusted.
📄 AOSP — AVB 2.0 · avbtool README · libavb source

④ dm-verity Setup — Layering on dm-linear

For each partition with a hashtree descriptor, fs_mgr creates a dm-verity target. The dm-verity mapping table looks like:

0 <data_size> verity 1 /dev/block/dm-0 /dev/block/dm-0 4096 4096 <data_blocks> <hash_offset> sha256 <root_hash> <salt>

Key fields:
· data device: the dm-linear node (e.g., /dev/block/dm-0)
· hash device: often the same dm-linear node (hash tree appended after data) or a separate partition
· root hash: from the vbmeta hashtree descriptor (signed by AVB)
· block size: 4096 bytes (both data and hash blocks)

The kernel creates a new dm node (e.g., /dev/block/dm-1) that wraps the dm-linear device. Every 4 KiB read is checked against the Merkle tree. If a block fails verification, dm-verity returns an I/O error (in eio mode) or triggers a device restart (in restart mode).

📄 AOSP — dm-verity · Kernel docs — dm-verity

⑤ Device Nodes — The Stacking Result

After both targets are created, the kernel exposes a stack of device nodes. For example, for system:

/dev/block/by-name/super → physical partition
/dev/block/dm-0 → dm-linear (carves system’s byte range)
/dev/block/dm-1 → dm-verity (verifies every read on dm-0)

The actual node numbers (dm-0, dm-1) are assigned dynamically. You can inspect them with dmsetup ls.

📄 dmsetup(8) man page

⑥ Mounting — Verified Partition Available

First-stage init mounts the dm-verity device (not the dm-linear device) at the mount point specified in fstab — e.g., /system, /vendor, /odm. The mount is read-only. From this point forward, every block read by any process passes through dm-verity transparently.

Userspace never interacts with dm-linear or dm-verity directly. To any process reading /system/framework/services.jar, it is a normal file read. The kernel device-mapper layer intercepts the block I/O and performs verification before returning data. This is why verified boot is “transparent” — no application changes are needed.

🎬 Animated Sequence — Carving & Verifying a Partition

Click Next to step through the five frames. Each frame shows one stage of the dm-linear → dm-verity flow for the system partition.

Frame 1 — Super partition layout. The physical super partition contains all dynamic partitions as contiguous byte ranges. LP metadata (parsed by liblp) stores the offset and size of each.

Frame 2 — Carve dm-linear target. fs_mgr reads LP metadata and creates a dm-linear mapping: 0 <sectors> linear /dev/block/by-name/super <start>. Result: /dev/block/dm-0 represents the system partition.

Frame 3 — AVB verification. libavb verifies the vbmeta RSA signature, checks the rollback index against RPMB, and extracts the root hash for each partition’s Merkle tree from the hashtree descriptor.

Frame 4 — dm-verity layering. A verity target is created with dm-0 as both data device and hash device. The trusted root_hash from AVB anchors the Merkle tree. Valid reads pass through; tampered blocks return EIO.

system: super → dm-linear (dm-0) → dm-verity (dm-1) → mount /system Same pattern applies to vendor, odm, product, system_ext

Frame 5 — Verified mount. First-stage init mounts the dm-verity device at /system (read-only). From this point, every file read on /system is transparently verified. The same stack is created for each dynamic partition.

✅ How to Verify — Developer Checklist

Commands to inspect dm-linear, dm-verity, and AVB state on a running device:

  • adb shell ls /dev/block/by-name/ | grep -E "super|vbmeta"Confirm super and vbmeta partitions exist as device nodes.
  • adb shell dmsetup lsList all device-mapper targets. Expect entries like system-verity, system (linear), vendor-verity, vendor. Each shows its dm-N node number.
  • adb shell dmsetup table systemShow the dm-linear mapping table. Expect: 0 <sectors> linear <super_dev> <start>.
  • adb shell dmsetup table system-verityShow the dm-verity mapping table. Expect: 0 <sectors> verity 1 <data_dev> <hash_dev> 4096 4096 <blocks> <hash_offset> sha256 <root_hash> <salt>.
  • adb shell dmsetup info system-verityShow dm target state. Check State: ACTIVE and Read Ahead: ....
  • adb shell cat /proc/mounts | grep dm-Verify mounted partitions use dm-verity device nodes (e.g., /dev/block/dm-1 /system ext4 ro...).
  • avbtool info_image --image vbmeta.imgInspect vbmeta contents on host. Shows algorithm, public key, descriptors, rollback index.
  • adb shell getprop ro.boot.verifiedbootstateCheck boot state: green = locked+verified, yellow = locked+custom-key, orange = unlocked.

⚠️ Troubleshooting — Common Failures

🚫 vbmeta Signature Invalid

vbmeta signed with wrong key or image modified after signing. Locked bootloader: red screen, boot halts. Unlocked: yellow warning, boots with verifiedbootstate=orange. Fix: re-sign vbmeta or re-flash official images.

🚫 Rollback Index Too Low

Attempting to flash an older image whose rollback_index is below the RPMB-stored minimum. Prevents downgrade attacks. Fix: update to newer image or (on unlocked devices) reset RPMB counter.

⚡ dm-verity Hash Mismatch (EIO)

A block on /system or /vendor has been modified. dm-verity returns I/O error in eio mode or reboots in restart mode. Diagnose: dmesg | grep verity. Fix: re-flash the partition or disable verity on unlocked device (fastboot --disable-verity).

⚡ dm-linear Target Creation Fails

LP metadata in super is corrupted or mismatched. fs_mgr logs errors like “Failed to create logical partition”. Fix: fastboot wipe super and re-flash.

🚫 Chained vbmeta Missing

Parent vbmeta references vbmeta_system or vbmeta_vendor but the partition is absent or unreadable. That partition group cannot mount. Fix: flash the missing vbmeta image.

⚡ Hash Tree Offset Wrong

Build-time mismatch between image size and hash tree offset in vbmeta descriptor. The dm-verity table points to wrong data for hash nodes. Diagnose: compare avbtool info_image output with actual image size. Fix: rebuild images with correct avbtool add_hashtree_footer.

🏭 Edge Cases & Device-Specific Notes

🔧 A/B Slot Considerations

On A/B devices, dynamic partitions inside super are duplicated by slot: system_a/system_b, vendor_a/vendor_b, etc. The LP metadata contains extent entries for both slots. First-stage init reads androidboot.slot_suffix from the kernel cmdline and only creates dm-linear/dm-verity targets for the active slot.

Similarly, vbmeta_a/vbmeta_b exist on A/B devices. AVB loads the vbmeta for the active slot. The rollback index is per-slot in RPMB.

🔧 When dm-verity Is Disabled

Unlocked bootloader + userdebug/eng builds: fastboot --disable-verity sets a flag in vbmeta that tells fs_mgr to skip dm-verity creation. dm-linear is still used (dynamic partitions still need carving), but the mount happens directly on the dm-linear device without verification.

Verity mode “none”: On some debug builds, dm-verity is created but in logging-only mode — errors are logged to dmesg but reads still succeed.

Uncertainty note: The exact mechanism for disabling verity varies by OEM. Some devices use avbctl disable-verification; others require specific vbmeta flags. Consult your device’s vendor documentation or avbctl source.

🔧 When dm-linear Is Not Used (Legacy Devices)

Devices launched before Android 10 (or upgraded devices that did not adopt dynamic partitions) have system, vendor, etc. as separate physical partitions. There is no super partition and no dm-linear layer. dm-verity is created directly on the physical partition device (e.g., /dev/block/by-name/system).

I do not know the exact list of devices that fall into this category for Android 16. Consult the device’s BoardConfig.mk for BOARD_SUPER_PARTITION_SIZE — if this variable is absent, the device likely uses legacy physical partitions.

❓ AVB Error Handling Modes

When dm-verity detects corruption, behaviour depends on the verity_mode flag:

  • eio — returns I/O error to the reading process (default for production)
  • restart — reboots the device when corruption is detected
  • none — logging only, no enforcement (debug builds)

I do not know whether Android 16 changes the default mode. Consult fs_mgr_avb.cpp and your device’s fstab for avb flag options.

# === On-device inspection ===

# List device-mapper targets (shows dm-linear and dm-verity)
adb shell dmsetup ls
# Example output:
#   system (253:0)
#   system-verity (253:1)
#   vendor (253:2)
#   vendor-verity (253:3)

# Show dm-linear mapping for system
adb shell dmsetup table system
# Example: 0 2457600 linear /dev/block/sda17 2048

# Show dm-verity mapping for system
adb shell dmsetup table system-verity
# Example: 0 2457600 verity 1 253:0 253:0 4096 4096 307200 307200 sha256 a3b2c1d4... 00000000...

# Check verified boot state
adb shell getprop ro.boot.verifiedbootstate   # green | yellow | orange

# Check dm-verity errors in kernel log
adb shell dmesg | grep -i verity

# === On-host vbmeta inspection ===

# Inspect vbmeta metadata
avbtool info_image --image vbmeta.img

# Verify vbmeta signature against a known key
avbtool verify_image --image vbmeta.img --key root_key.pem

# Create vbmeta with hashtree descriptor
avbtool make_vbmeta_image \
  --include_descriptors_from_image system.img \
  --key /path/to/signing_key.pem \
  --algorithm SHA256_RSA4096 \
  --rollback_index 1 \
  --output vbmeta.img
