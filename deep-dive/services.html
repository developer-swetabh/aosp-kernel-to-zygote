🚀 Second-Stage init — Service Startup

Parses the full init.rc tree, starts services by class in dependency order, and prepares the system for Zygote launch.

📋 Overview

  • Second-stage init: After SELinux transition, init re-parses the full init.rc tree with access to all mounted partitions.
  • Service classes: Services are organized into classes (core, main, late_start) that start in defined order.
  • Property triggers: Many services start in response to property changes (e.g., on property:sys.boot_completed=1).

⚙️ How It Works

Parse Full init.rc Tree

Second-stage init reads init.rc and all imported *.rc files from /system/etc/init/, /vendor/etc/init/, /odm/etc/init/. Each file defines services, triggers, and actions.

Start Property Service

The property service (property_service) starts — it manages setprop/getprop calls. Properties drive trigger-based service startup.

Fire Triggers in Order

Triggers fire in sequence: early-init → init → late-init → post-fs → post-fs-data → boot. Each trigger runs its associated action list.

Start Class core

Essential system daemons: ueventd (device node manager), logd (logging), servicemanager (Binder registry), hwservicemanager (HIDL registry).

Start Class main

Core system services: vold (volume daemon), netd (network daemon), surfaceflinger (display compositor), and zygote.

Start Class late_start

Non-critical services that start after sys.boot_completed=1. These include optional vendor services, analytics daemons, and background tasks.

🔗 Service Class Startup Sequence

🌲 init.rc Parsing & Trigger Sequence

Second-stage init parses the full init.rc tree by importing all *.rc files from mounted partitions, then fires triggers in a defined order. This is exclusively a second-stage responsibility — first-stage init never reads init.rc.

📁 Key Files & Artifacts

FilePurpose
init.rcMain init configuration — defines triggers, actions, and service imports
/system/etc/init/*.rcSystem service definitions (zygote, surfaceflinger, logd, etc.)
/vendor/etc/init/*.rcVendor/HAL service definitions
/odm/etc/init/*.rcODM-specific service definitions
servicemanagerBinder service registry — the central hub for IPC
hwservicemanagerHIDL service registry for hardware abstraction layers

⚠️ Common Checks & Failure Modes

🚫 Service Crash Loop

If a service crashes repeatedly, init may either restart it (with backoff) or give up and set a failure property. Check logcat for crash dumps.

🚫 servicemanager Not Ready

If servicemanager fails to start, no Binder services can register. All dependent services (including system_server) will fail.

⚡ Missing HAL

A service depending on a HAL that isn't registered in hwservicemanager will hang or crash. Check lshal output.

💻 Code Snippets

init.rc 
# Example init.rc trigger and service definition (second-stage)
on early-init
    mount tmpfs tmpfs /dev mode=0755

on init
    mkdir /mnt/vendor/persist 0771 root system

on late-init
    trigger post-fs
    trigger boot

on boot
    class_start core
    class_start main

on property:sys.boot_completed=1
    class_start late_start

service zygote /system/bin/app_process64 -Xzygote /system/bin \
    --zygote --start-system-server
    class main
    priority -20
    user root
    group root readproc reserved_disk
    socket zygote stream 660 root system
bash 
# List all init services and their status
adb shell getprop | grep init.svc

# Check a specific service
adb shell getprop init.svc.zygote    # running, stopped, restarting

# List registered Binder services
adb shell service list

# List registered HAL services
adb shell lshal

# View service startup timing
adb shell dmesg | grep "init: starting service"
