🧬 Zygote — Android Runtime Origin

The process that births every Android app — preloads the framework, starts SystemServer, then listens to fork new app processes on demand.

📋 Overview

  • Origin process: Zygote is started by init as app_process64 (or app_process32). It is the ancestor of every Android application process.
  • Class preloading: Preloads ~8000 framework classes and shared resources (drawables, layouts) into memory. These pages are shared across all forked processes via copy-on-write (COW).
  • Fork model: New app processes are created by fork() from Zygote, not exec(). This makes app startup extremely fast (~200ms) because the entire framework is already in memory.

⚙️ How It Works

init Starts Zygote

Init starts the zygote service defined in init.zygote64.rc. The binary is /system/bin/app_process64 with arguments --zygote --start-system-server.

ZygoteInit.main()

The app_process binary initializes the Android Runtime (ART) and calls ZygoteInit.main() in Java. This is where the Java world begins.

Preload Classes & Resources

Preloads ~8000 classes from /system/etc/preloaded-classes, shared libraries, and common resources. These become shared memory pages for all future apps.

Fork SystemServer

Zygote forks its first child: system_server. SystemServer starts 100+ framework services: ActivityManagerService (AMS), PackageManagerService (PMS), WindowManagerService (WMS), etc.

Open Zygote Socket

Zygote creates a UNIX domain socket (/dev/socket/zygote) and enters runSelectLoop(). It now waits for fork requests from system_server (via AMS).

Fork App Processes

When a user launches an app, AMS sends a request to the Zygote socket. Zygote calls fork() + Zygote.specialize() to create the new process with the correct UID, SELinux context, and capabilities.

⚠️ Common Checks & Failure Modes

🚫 Zygote Crash

If Zygote crashes, all app processes die (they're children of Zygote). Init restarts Zygote, which restarts system_server, causing a full soft reboot.

🚫 system_server Crash

system_server crash triggers a Zygote restart (system_server is critical). Results in "System UI isn't responding" dialog and a soft reboot.

⚡ Preload Failure

If a preloaded class is missing or corrupted, Zygote may crash during startup. Check logcat for class loading errors.

⚡ Socket Permission Denied

If SELinux denies access to the Zygote socket, AMS can't request app forks. Apps won't start. Check avc: denied for zygote socket.

🏭 Vendor-Specific Details

🔧 32-bit vs 64-bit Zygote

On 64-bit devices, two Zygote processes may run: zygote (64-bit, primary) and zygote_secondary (32-bit, for legacy apps). Defined in init.zygote64_32.rc.

The 32-bit Zygote uses app_process32 and has its own socket (/dev/socket/zygote_secondary).

🔧 USAP (Unspecialized App Process) Pool

Since Android 10, Zygote can maintain a pool of pre-forked "unspecialized" processes. When an app launch request arrives, an existing USAP is specialized (faster than fork).

Controlled by persist.device_config.runtime_native.usap_pool_enabled.

💻 Code Snippets

bash 
# Check Zygote process
adb shell ps -ef | grep zygote
# root  300  1  ...  zygote64
# root  301  1  ...  zygote (32-bit secondary)

# Check system_server
adb shell ps -ef | grep system_server
# system  500  300  ...  system_server

# View Zygote socket
adb shell ls -lZ /dev/socket/zygote
# srw-rw----  root system u:object_r:zygote_socket:s0  zygote

# Count preloaded classes
adb shell wc -l /system/etc/preloaded-classes
# ~8000

# Check boot completion
adb shell getprop sys.boot_completed   # 1
init.zygote64.rc 
service zygote /system/bin/app_process64 -Xzygote /system/bin \
    --zygote --start-system-server --socket-name=zygote
    class main
    priority -20
    user root
    group root readproc reserved_disk
    socket zygote stream 660 root system
    socket usap_pool_primary stream 660 root system
    onrestart exec_background -- /system/bin/vdc volume abort_fuse
    onrestart write /sys/power/state on
    onrestart restart audioserver
    onrestart restart cameraserver
    onrestart restart media
    onrestart restart --only-if-running netd
    task_profiles ProcessCapacityHigh MaxPerformance
    critical window=${zygote.critical_window.minute:-off} \
        target=zygote-fatal
