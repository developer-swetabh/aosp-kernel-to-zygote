📜 init — First Stage (Ramdisk)

The first userspace process (PID 1) — parses init.rc, reads fstab, and triggers AVB verification and early mounts.

📋 Overview

  • Identity: /init runs as PID 1, the ancestor of all userspace processes. It never exits.
  • Two stages: First-stage init runs from ramdisk (before real root); second-stage init runs after filesystems are mounted.
  • Key action: Reads fstab.<hw> to discover partitions, triggers AVB, and performs early mounts of /system, /vendor, etc.

⚙️ How It Works

Kernel Executes /init

Kernel calls run_init_process("/init"). This is the first-stage init binary, linked statically, running from the ramdisk tmpfs.

Parse fstab.<hw>

First-stage init reads /fstab.qcom (or /vendor/etc/fstab.<hw>) to discover the partition layout, mount flags, and filesystem types. This file maps device paths to mount points.

Trigger AVB Verification

Calls fs_mgr to set up AVB handles. Verifies vbmeta chain for partitions listed in fstab with avb flag. Sets up dm-verity and dm-linear targets.

Early Mounts

Mounts /system, /vendor, /odm, /product as read-only. These are needed before second-stage init can load SELinux policy and additional configs.

Load SELinux Policy

Loads the compiled sepolicy from the mounted partitions. Transitions init to the u:r:init:s0 SELinux context.

Re-execute as Second-Stage init

Calls execv("/init") with a flag indicating second stage. Second-stage init parses the full init.rc tree and begins starting services.

⚠️ Common Checks & Failure Modes

🚫 fstab Not Found

If fstab.<hw> is missing or malformed, init cannot discover partitions. Boot stalls before mounting /system.

🚫 init.rc Syntax Error

A syntax error in any imported *.rc file causes the entire section to be skipped. Check dmesg for "init: /init.rc: ..." error messages.

⚡ SELinux Denial on Mount

If SELinux contexts don't match, mount operations may be denied even if the partition exists. Check adb logcat | grep avc.

⚡ Missing Device Node

If a block device referenced in fstab doesn't appear (driver not loaded), mount fails. First-stage init waits with a timeout.

🏭 Vendor-Specific Details

🔧 Qualcomm fstab.qcom

Qualcomm devices use fstab.qcom. Partition paths typically reference /dev/block/bootdevice/by-name/ symlinks created by the QTI uevent daemon.

Qualcomm adds custom fstab flags: latemount, wait, slotselect.

🔧 First-stage vs Second-stage init

First-stage: Runs from ramdisk. Handles AVB, early mounts, SELinux load. Minimal environment — no properties system, no service manager.

Second-stage: Runs after execv("/init") with full filesystem. Starts property service, parses all init.rc imports, begins service management.

💻 Code Snippets

init.rc 
# Example init.rc trigger and service definition
on early-init
    mount tmpfs tmpfs /dev mode=0755

on init
    mkdir /mnt/vendor/persist 0771 root system

on late-init
    trigger post-fs
    trigger boot

service zygote /system/bin/app_process64 -Xzygote /system/bin \
    --zygote --start-system-server
    class main
    priority -20
    user root
    group root readproc reserved_disk
    socket zygote stream 660 root system

# Qualcomm fstab example (vendor-specific)
