📜 First-Stage init (Ramdisk)

The very first userspace process (PID 1), running from the ramdisk before real partitions exist — responsible for creating the device environment, mounting critical filesystems, and handing off to second-stage init.

📋 Overview

  • Identity: /init is executed by the kernel as PID 1 — the ancestor of every userspace process. It is a statically linked binary running from the ramdisk tmpfs, because no real filesystem is mounted yet. (source: first_stage_init.cpp)
  • Scope: First-stage init handles only the minimum work needed to make real partitions available: virtual FS mounts, device nodes, fstab parsing, dm-verity/dm-linear setup, and early partition mounts. It does not parse init.rc, start services, or manage properties — those are second-stage responsibilities.
  • Exit: First-stage init ends by calling execv("/system/bin/init") (or re-executing /init with a second-stage flag) to replace itself with second-stage init, preserving PID 1.

🎭 Think of It This Way

Imagine a theater stage crew working before the curtain rises. They arrive in a dark, empty theater (the ramdisk). Their job is to: turn on the lights (mount /dev, /proc, /sys), plug in the microphones (start ueventd to create device nodes), read the seating chart (parse fstab), check safety equipment (set up AVB / dm-verity), position the props (mount /system, /vendor), and finally hand the stage to the director (call execv() to launch second-stage init), who cues the actors (services) and opens the show.

The crew never reads the script (init.rc) — that is the director's job. The crew only prepares the stage.

⚙️ How It Works — Seven First-Stage Responsibilities

① Mount Virtual Filesystems (/dev, /proc, /sys)

The kernel starts /init in a bare ramdisk with almost nothing mounted. First-stage init immediately mounts the essential virtual filesystems:

• mount("tmpfs", "/dev", "tmpfs") — writable /dev for device nodes.
• mount("proc", "/proc", "proc") — kernel state (process info, cmdline).
• mount("sysfs", "/sys", "sysfs") — kernel device model and driver info.
Also mounts /dev/pts (pseudo-terminals) and /sys/fs/selinux.

📄 first_stage_init.cpp — FirstStageInit()

② Start ueventd to Populate /dev

After mounting /dev, it is empty — no block device nodes exist yet. First-stage init forks ueventd, which listens for kernel uevent messages and creates device nodes under /dev/block/. This is critical because partition mounts need paths like /dev/block/by-name/system.

ueventd reads ueventd.rc from the ramdisk for ownership/permissions and creates platform-specific symlinks (e.g., /dev/block/bootdevice/by-name/ on Qualcomm).

Without ueventd, fstab entries referencing /dev/block/by-name/<partition> resolve to nothing, and all mount operations fail with "no such device."
📄 ueventd.cpp · AOSP — ueventd

③ Read and Apply fstab Entries

First-stage init reads the device fstab to discover partitions, mount points, and filesystem-manager flags. Location varies by device:

• Ramdisk: /fstab.<hw> (legacy) or /first_stage_ramdisk/fstab.<hw>
• Vendor ramdisk: /vendor/etc/fstab.<hw> (modern GKI devices)
• Device-tree: /proc/device-tree/firmware/android/fstab (deprecated)

Only entries with the first_stage_mount flag are processed. Other entries are deferred to second-stage init or vold.

The <hw> in fstab.<hw> is vendor-specific. Qualcomm → fstab.qcom; Tensor → fstab.tensor; Exynos → fstab.exynos. Determined by androidboot.hardware in the kernel command line. Run cat /proc/cmdline to check.
📄 first_stage_mount.cpp — ReadFstab() · AOSP — fstab

④ Handle A/B Slot Switching Logic

On A/B devices, partitions are duplicated (system_a / system_b). First-stage init resolves the active slot:

• Reads slot suffix from kernel cmdline (androidboot.slot_suffix=_a) or bootconfig.
• For fstab entries with slotselect, appends the suffix (e.g., system → system_a).
• All subsequent dm-verity and mount operations use the slot-resolved partition names.

The bootloader already selected the slot — first-stage init merely reads that decision and applies it to mount paths.

A/B slot awareness is essential because dm-verity hash trees and vbmeta descriptors are slot-specific. Mounting system_b's image with system_a's vbmeta would fail AVB verification.
📄 AOSP — A/B Slots · fs_mgr_fstab.cpp

⑤ Create Device-Mapper Targets (dm-linear, dm-verity)

Before mounting, first-stage init uses the kernel device-mapper framework to set up two critical layers:

dm-linear: On devices with dynamic partitions, logical partitions are carved from the physical super partition. fs_mgr reads the partition table from super and creates dm-linear targets mapping each logical partition to its byte range.

dm-verity: For partitions with the avb fstab flag, fs_mgr reads hash-tree descriptors from vbmeta and configures dm-verity. This creates a verified device-mapper node that checks each 4 KiB block against the Merkle tree on read.

The resulting nodes (e.g., /dev/block/dm-0) are what get mounted — not the raw partition devices.

The dm-verity setup here is what makes Android Verified Boot work at runtime. Without it, a tampered /system would go undetected. The hash tree root signed in vbmeta anchors the chain of trust.
📄 AOSP — dm-verity · AOSP — Dynamic Partitions · first_stage_mount.cpp

⑥ Mount Vendor and Platform Partitions

With dm targets ready, first-stage init mounts partitions flagged first_stage_mount:

• /system — Android framework and core binaries (often mounted at / via system-as-root)
• /vendor — HAL libraries, vendor init scripts, firmware
• /odm — ODM customizations
• /product, /system_ext — optional, vary by device

All mounts are read-only. These must be available before second-stage init because it needs SELinux policy (from /vendor/etc/selinux/) and the init.rc tree (from /system/etc/init/).

📄 AOSP — Partitions and Images · first_stage_mount.cpp — DoFirstStageMount()

⑦ execv() Handoff to Second-Stage init

Once early mounts are complete, first-stage init performs the handoff:

1. Loads compiled SELinux policy from newly-mounted partitions; transitions to enforcing mode.
2. Re-labels /dev, /proc, /sys with correct SELinux contexts.
3. Calls execv("/system/bin/init", {"second_stage", ...}).

The execv() call replaces the current process image — the same PID 1 now runs the second-stage init binary. This is not a fork; first-stage init ceases to exist. Second-stage init then parses the full init.rc tree, starts the property service, fires triggers, and begins launching services. See Second-Stage init — Service Startup for that flow.

Using execv() instead of fork() preserves PID 1 — Android requires init to be PID 1 for its entire lifetime. It also cleanly separates the minimal ramdisk environment from the fully-mounted one. This is the "curtain rise" moment.

✅ How to Verify — Developer Checklist

Use these commands on a running device to confirm first-stage init completed successfully:

  • adb shell ls /dev/block/by-name/Confirm device nodes exist. Empty → ueventd failed or storage drivers missing.
  • adb shell cat /proc/cmdline | tr ' ' '\n' | grep androidbootCheck androidboot.slot_suffix, .hardware, .verifiedbootstate.
  • adb shell cat /proc/mounts | grep -E "system|vendor|odm"Verify critical partitions are mounted. Check for dm- device paths (dm-verity active).
  • adb shell ls /dev/block/dm-*Confirm device-mapper targets were created.
  • adb shell ps -p 1 -o comm=Should return init. PID 1 survived the execv handoff.
  • adb shell getenforceShould return Enforcing. SELinux loaded before execv.
  • adb shell dmesg | grep -i "first stage"First-stage log messages. Useful for diagnosing mount/fstab errors.

⚠️ Common Pitfalls & Failure Modes

🚫 fstab Not Found or Malformed

Missing or syntax-broken fstab.<hw> → init cannot discover partitions. Boot stalls before /system mount. Verify androidboot.hardware matches the fstab filename.

🚫 dm-verity Setup Failure

vbmeta hash mismatch after flashing a modified system image → dm-verity creation fails. Locked device: boot aborts. Unlocked: warning. Fix by re-flashing vbmeta or fastboot --disable-verity.

⚡ Missing Device Node (ueventd timeout)

Block device not in /dev/block/ (storage driver issue, wrong DTB) → init waits and then panics. Check dmesg for storage driver errors.

⚡ Wrong Partition Flashed to Wrong Slot

system_a image in slot B → slot suffix resolution mounts wrong data. dm-verity catches the hash mismatch. Always verify fastboot getvar current-slot before flashing.

🚫 SELinux Policy Load Failure

If /vendor/etc/selinux/ is missing (mount failed), policy cannot load and execv to second-stage fails. Root cause is usually step ⑥.

⚡ Dynamic Partition Table Corruption

Corrupted super metadata → dm-linear fails. Recovery: fastboot wipe super and re-flash.

🏭 Vendor-Specific Details

🔧 Qualcomm — fstab.qcom and bootdevice symlinks

Qualcomm uses fstab.qcom. Paths reference /dev/block/bootdevice/by-name/ symlinks from QTI uevent rules. Common flags: wait, slotselect, logical, first_stage_mount, avb=vbmeta. If /dev/block/bootdevice/ is missing, the platform device driver may not have probed.

🔧 fstab location — ramdisk vs vendor_boot vs DTB

Android 9–10: /fstab.<hw> in ramdisk root.

Android 11+: /vendor/etc/fstab.<hw> (vendor_boot ramdisk). Also checks /first_stage_ramdisk/fstab.<hw>.

DTB fstab: /proc/device-tree/firmware/android/fstab/ — deprecated but supported.

Uncertainty note: Exact lookup order varies by AOSP version. Consult first_stage_mount.cpp — ReadFstab() for your target.

🔧 System-as-root vs ramdisk-as-root

System-as-root (Android 10+): system partition mounted at /. Ramdisk only has /init, fstab, ueventd.rc.

Legacy (Android 9−): Ramdisk is root; /system is a separate mount point.

Android 13+ devices use init_boot.img for the generic ramdisk. See Boot Image Format.

❓ Recovery mode handling

On A/B devices using recovery-as-boot, first-stage init checks for androidboot.force_normal_boot=1. If absent, it launches recovery. Uncertainty: Devices with a dedicated recovery partition may differ. Check BOARD_USES_RECOVERY_AS_BOOT in BoardConfig.mk.
