🔒 SELinux / sepolicy Setup

Loads mandatory access control policy, labels all filesystems, and transitions init to enforcing mode — the security foundation for all of Android.

📋 Overview

  • MAC enforcement: SELinux provides mandatory access control — even root processes are constrained by policy. Every process, file, and socket has a security context.
  • Policy sources: Policy fragments (CIL) from /system and /vendor are compiled at build time into a single binary policy.
  • Init transition: After loading policy, init re-executes itself in the u:r:init:s0 domain, beginning second-stage init with full SELinux enforcement.

⚙️ How It Works

Mount selinuxfs

First-stage init mounts /sys/fs/selinux (selinuxfs). This kernel pseudo-filesystem is the interface between userspace and the kernel's SELinux subsystem.

Load Compiled Policy

Load the precompiled binary policy from /vendor/etc/selinux/precompiled_sepolicy (or /system/etc/selinux/). The policy was compiled from CIL fragments at build time using secilc.

Set Enforcing Mode

Write 1 to /sys/fs/selinux/enforce. From this point, all access decisions are enforced. Violations result in denied operations (logged as avc: denied).

Label Filesystems (restorecon)

Run restorecon -R on all mounted filesystems to apply file security contexts from file_contexts. This ensures every file has the correct SELinux label.

Re-execute init in SELinux Domain

Init calls execv("/init") with selinux_android_restorecon("/init"). The kernel transitions init to u:r:init:s0 — the proper domain for second-stage init.

📁 Key Files & Artifacts

FilePurpose
precompiled_sepolicyBinary SELinux policy loaded by init at boot
file_contextsMaps file paths (regex) to SELinux security contexts
property_contextsMaps Android system properties to SELinux contexts
service_contextsMaps Binder service names to SELinux contexts
/sys/fs/selinux/enforceKernel interface — read/write enforcing mode (1 = enforce, 0 = permissive)
seapp_contextsMaps app UID/package to SELinux domain at process creation

⚠️ Common Checks & Failure Modes

🚫 Policy Version Mismatch

If system and vendor policy fragments are from different Android versions, secilc compilation may fail or produce a policy that the kernel rejects.

🚫 avc: denied (Enforcing)

A process tries an action not allowed by policy. The operation is blocked. Check adb logcat | grep avc for the denial details.

⚡ restorecon Failure

If file_contexts patterns don't match actual file paths, files keep incorrect labels. Services may fail to access them.

💻 Code Snippets

bash 
# Check SELinux status
adb shell getenforce    # Enforcing or Permissive

# View security context of a process
adb shell ps -eZ | grep init
# u:r:init:s0  root  1  0  init

# Check file security context
adb shell ls -Z /system/bin/app_process64

# Analyze AVC denials
adb logcat | grep "avc: denied"

# Convert denial to allow rule (dev only)
adb logcat | grep avc | audit2allow -p policy
