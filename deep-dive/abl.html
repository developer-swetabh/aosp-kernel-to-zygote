🔐 ABL — Android Bootloader

The first software stage after SoC ROM — verifies boot integrity, selects the active slot, and loads the kernel.

📋 Overview

  • Purpose: Bridge between SoC ROM and the Linux kernel — the last vendor-controlled boot stage before AOSP code runs.
  • Key responsibility: Verify the boot chain using Android Verified Boot (AVB) signatures stored in vbmeta.
  • Output: Loads the kernel + ramdisk into memory and transfers control by jumping to the kernel entry point.

⚙️ How It Works

Power-On Reset (POR)

SoC hardware executes immutable ROM code (PBL — Primary Boot Loader). PBL loads the SoC-specific bootloader chain (e.g., Qualcomm: PBL → SBL → XBL → ABL).

Load ABL from Boot Partition

ABL binary is read from the abl partition (or embedded in XBL on some SoCs). It initializes display, USB, and storage drivers needed for the next stages.

Read and Verify vbmeta

ABL reads vbmeta.img from the vbmeta partition. It verifies the RSA-4096 or RSA-2048 signature against an embedded root-of-trust public key. If the signature is invalid, boot is halted (locked device) or a warning is shown (unlocked device).

A/B Slot Selection

Reads boot control metadata from the misc partition. Evaluates active, bootable, successful_boot, and retry_count flags for each slot (A/B). Selects the highest-priority bootable slot.

Load Kernel + Ramdisk

Reads the boot.img (or init_boot.img since Android 13) from the selected slot. Decompresses the kernel image and initial ramdisk into RAM. Sets up kernel command-line parameters (androidboot.*) and jumps to the kernel entry point.

⚠️ Common Checks & Failure Modes

🚫 vbmeta Signature Invalid

On locked devices: boot halts, device shows red warning screen. On unlocked devices: yellow warning shown, boot continues.

🚫 Both Slots Unbootable

If both A and B slots are marked unbootable (retry_count exhausted), device enters recovery/fastboot mode.

⚡ Rollback Protection Triggered

AVB rollback index in vbmeta is lower than the stored minimum — prevents downgrade attacks. Boot is refused.

⚡ boot.img Hash Mismatch

The hash of the loaded boot image doesn't match the descriptor in vbmeta. Indicates corruption or tampering.

🏭 Vendor-Specific Details

🔧 Qualcomm (XBL → ABL chain)

Qualcomm SoCs use a multi-stage bootloader: PBL → SBL1 → XBL (UEFI) → ABL. XBL initializes DDR, clocks, and storage. ABL runs as a UEFI application within XBL's environment.

Key partitions: xbl, xbl_config, abl, devcfg.

ABL source: bootable/bootloader/edk2 in Qualcomm BSP, built as a UEFI DXE driver.

🔧 MediaTek (preloader → LK)

MediaTek uses preloader → LK (Little Kernel). The preloader initializes DRAM and loads LK, which acts as the Android bootloader.

LK handles AVB verification and slot selection similarly to ABL on Qualcomm.

🔧 Google Tensor

Tensor SoCs (Pixel 6+) use a custom bootloader chain based on Samsung Exynos firmware. The ABL equivalent handles AVB and slot selection in a proprietary binary.

💻 Code Snippets

bash 
# Verify vbmeta signature using avbtool
avbtool verify_image --image vbmeta.img --key /path/to/root_key.pem

# Extract vbmeta info (hash descriptors, flags, rollback index)
avbtool info_image --image vbmeta.img

# Check A/B slot status via fastboot
fastboot getvar current-slot
fastboot getvar slot-retry-count:a
fastboot getvar slot-successful:a
fastboot getvar slot-unbootable:a
text 
# Example kernel command line set by ABL:
androidboot.verifiedbootstate=green
androidboot.slot_suffix=_a
androidboot.vbmeta.device_state=locked
androidboot.vbmeta.hash_alg=sha256
androidboot.vbmeta.size=8192
androidboot.vbmeta.invalidate_on_error=yes
