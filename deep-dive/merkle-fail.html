🌳 Merkle Hash Tree & 🚫 AVB Failure Branch

How dm-verity builds and verifies the Merkle hash tree at read time — and what happens when verification fails.

📋 Overview

  • Merkle tree verification: dm-verity uses a binary hash tree where each leaf is the SHA-256 of a 4 KiB data block. The root hash is signed in vbmeta.
  • Lazy verification: Blocks are verified on read, not at boot — no full partition scan needed. This keeps boot fast.
  • Failure path: If AVB or Merkle verification fails, the device halts (locked), shows warning (unlocked), or enters recovery — depending on error mode.

⚙️ How Merkle Verification Works

Read hashtree_descriptor from vbmeta

The hashtree_descriptor contains: partition name, hash algorithm (SHA-256), salt, root hash, tree offset, and tree size. This descriptor is signed as part of vbmeta.

Build In-Memory Merkle Tree Structure

dm-verity creates the tree metadata in memory. Each leaf node covers one 4 KiB data block. Parent nodes are the hash of their children's hashes concatenated. The tree depth depends on partition size.

Block Read Triggers Verification

When a process reads a block, dm-verity intercepts the I/O. It computes SHA-256(salt + block_data) and walks up the tree, verifying each level matches until reaching the root.

Root Hash Check

The computed root hash is compared against the root hash from hashtree_descriptor. If it matches, the read proceeds. If it doesn't, corruption is detected.

Corruption Response

Based on error mode: eio = return I/O error to the calling process; restart = reboot device; none = log only (debug).

🚫 AVB Failure Path

🔒 Locked Device + Verification Fail

Boot is halted immediately. A red warning screen is displayed. The device cannot proceed past the bootloader. User must reflash with verified images.

🔓 Unlocked Device + Verification Fail

A yellow/orange warning screen is shown for 5 seconds. Boot continues with androidboot.verifiedbootstate=orange. dm-verity may run in logging mode.

⚡ Runtime dm-verity EIO

A block read fails Merkle verification at runtime. The reading process receives an I/O error (-EIO). System may become unstable if critical blocks are affected.

⚡ Slot Marked Unbootable

After repeated AVB failures, the boot control HAL marks the slot as unbootable. The bootloader switches to the other slot on next boot.

🏭 Vendor-Specific Details

🔧 dm-verity Error Handling Modes

The verity_mode kernel parameter (set by fstab or bootloader) controls behavior on corruption:

  • eio — Return I/O error. Affected files become unreadable. (Production default)
  • restart — Reboot the device. More aggressive but ensures clean state.
  • none — Log only, allow read to succeed. (Debug/eng builds)
🔧 Hash Tree Storage

The hash tree data is appended to the partition image itself. The hashtree_descriptor in vbmeta tells dm-verity where the tree starts (offset) and its size.

For a 2 GB system partition with 4 KiB blocks, the hash tree adds approximately 32 MB of overhead (~1.5%).
